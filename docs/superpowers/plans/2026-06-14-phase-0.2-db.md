# Phase 0.2 — DB Layer Implementation Plan

> **For agentic workers:** implemented via superpowers:subagent-driven-development, with parallel waves for independent modules. Architecture is approved in the master spec (`docs/superpowers/specs/2026-06-13-podkit-platform-design.md`, §4 DB). This plan details the build.

**Goal:** A capable DB pillar: schema-as-code in TS, versioned migration files, generate + apply migrations against real Postgres, `db pull` (introspect live DB → migration files + reconcile), two-tier RLS (helpers + custom), all driven by `podkit db ...`, with an embedded Postgres (pglite) so local = cloud.

**Composed engines (the "no layer lacks" strategy):**
- **drizzle-orm** — typed query client + schema-as-code (`pgTable`, column types).
- **drizzle-kit** — migration `generate` (schema diff → SQL) and `pull`/introspect.
- **@electric-sql/pglite** — real in-process Postgres (WASM) for local dev and tests; no external DB required.

**New package:** `@podkit/db` at `packages/db`. Consumed later by `@podkit/cli` (the `db` command) and `@podkit/framework` (RLS-aware client injection — the auth-session wiring lands in Phase 0.3).

**Conventions (unchanged):** ESM, explicit `.ts`/`.tsx` import extensions, erasable-only TypeScript (no parameter properties/enums/namespaces — Node strips types), Node v26, pnpm, tests via `pnpm vitest run <path>`.

## File structure

```
packages/db/
  package.json
  src/
    schema.ts          # podkit schema conventions: re-export drizzle pg-core + helpers
    client.ts          # createDbClient(): pglite + drizzle, typed db + raw() + close()
    migrations/
      files.ts         # migration file naming/list/read (pure, timestamp injected)
      generate.ts      # schema diff → SQL migration file (drizzle-kit)  [SEQUENTIAL]
      apply.ts         # apply pending migration files to a db          [SEQUENTIAL]
    pull.ts            # introspect live DB → migration file + schema reconcile [SEQUENTIAL]
    rls/
      policy.ts        # RLS DSL → CREATE POLICY SQL (helpers + custom)
    index.ts           # public exports
  test/
    schema.test.ts
    client.test.ts
    files.test.ts
    rls.test.ts
    generate.test.ts
    apply.test.ts
    pull.test.ts
packages/cli/src/commands/db.ts   # `podkit db migrate|pull|studio`   [SEQUENTIAL]
examples/hello/app/db/schema.ts   # example: a `posts` table
```

**Task A1:** Create `packages/db` package; add deps `drizzle-orm`, `@electric-sql/pglite`, dev-dep `drizzle-kit`; `pnpm install`; confirm the workspace resolves. Commit `chore(db): scaffold @podkit/db package`.

## Wave B — independent modules (PARALLEL — disjoint files, unit-tested in isolation, no drizzle-kit)

Each is built TDD-first, in its own file, with no cross-dependency. Implementers write + test but do NOT commit (coordinator commits sequentially after the wave to avoid git-index races).

**Task B1 — Schema conventions** (`src/schema.ts` + `test/schema.test.ts`): re-export drizzle `pg-core` table/column builders under a podkit-stable surface, plus a `uuidPk()` helper (uuid primary key with default). Test: define a `posts` table and assert its inferred columns/types via drizzle's table introspection.

**Task B2 — DB client** (`src/client.ts` + `test/client.test.ts`): `createDbClient(opts: { dataDir?: string })` → returns `{ db, raw(sql), close() }` where `db` is a drizzle client over a pglite instance (in-memory when no `dataDir`). Test (in-memory): `raw("create table t(id int)")`, insert via `raw`, select via `raw`, assert rows; `close()` cleans up. (Real Postgres semantics through pglite.)

**Task B3 — Migration file utilities** (`src/migrations/files.ts` + `test/files.test.ts`): pure functions — `migrationFilename(timestamp: number, name: string)` → `0000000000000_name.sql` style (zero-padded, sanitized name); `listMigrations(dir)` → sorted `{ id, name, path }[]`; `readMigration(path)` → SQL string. Timestamp is a parameter (keeps it pure/testable). Test against a temp dir fixture.

**Task B4 — RLS policy DSL** (`src/rls/policy.ts` + `test/rls.test.ts`): pure SQL generation. `ownedBy(table, column, opts?)`, `inOrg(table, column)`, `isAgent(table)` convenience helpers, and `customPolicy(table, name, using, opts?)` escape hatch → each returns a `CREATE POLICY ...` (+ `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`) SQL string keyed off session GUCs (`current_setting('podkit.user_id')`, `podkit.org_id`, `podkit.is_agent`). Test the emitted SQL strings exactly for each helper + a custom policy.

---

## Wave C — integration (SEQUENTIAL, after Wave B; the drizzle-kit-dependent parts)

**Task C1 — Generate** (`src/migrations/generate.ts`): given a schema module path + migrations dir, produce a new SQL migration file capturing the diff (drizzle-kit). Validate the programmatic vs shell-out path; if drizzle-kit's API is awkward, shell out to the `drizzle-kit generate` CLI with a generated config. Test: generate from an empty state for the example `posts` schema → a migration file containing `CREATE TABLE`.

**Task C2 — Apply** (`src/migrations/apply.ts`): apply pending migration files (from `files.ts`) to a db (from `client.ts`) in order, tracking applied ids in a `_podkit_migrations` table. Test against in-memory pglite: apply the generated migration → table exists; re-apply is a no-op (idempotent).

**Task C3 — Pull** (`src/pull.ts`): introspect a live db → emit a migration file representing current DDL AND reconcile into the TS schema (drizzle-kit introspect/pull). Test: create a table out-of-band via `raw`, run pull, assert a migration file + schema reflect it. (If drizzle-kit introspect can't target pglite, document the constraint and target a real Postgres URL via env, keeping the test gated.)

**Task C4 — CLI wiring** (`packages/cli/src/commands/db.ts` + register in `bin.ts`): `podkit db migrate` (generate+apply), `podkit db pull`, `podkit db studio` (stub returning `E_*` "not yet"), all returning the `--json` envelope with structured errors. Example schema at `examples/hello/app/db/schema.ts`. Smoke: `podkit db migrate --json` in the example app.
